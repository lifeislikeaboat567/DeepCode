# DeepCode 0.2 项目总览与面试指南

## 1. 项目定位

DeepCode 是一个面向软件工程场景的大模型应用项目，目标不是单纯做“聊天”，而是把大模型能力组织成一个可执行、可治理、可扩展、可接入外部平台的工程智能体系统。项目同时提供 CLI、REST API、Reflex Web UI、平台消息桥接与扩展注册能力，覆盖从本地开发到平台接入的完整链路。

一句话定义：

> DeepCode 是一个支持多智能体协作、工具调用、运行时治理、平台桥接与可视化运维的 AI 软件工程助手。

## 2. 项目解决了什么问题

传统 LLM Demo 常见问题：

- 只有聊天，没有任务闭环
- 只有 prompt，没有工程化状态管理
- 只有单轮问答，没有会话、任务、产物沉淀
- 只有 API 调用，没有运维与治理视角
- 只能在网页里使用，无法接入企业 IM 或自动化通道

DeepCode 0.2 针对这些问题做了工程化落地：

- 用多智能体拆分规划、编码、评审、测试等职责
- 用工具路由把 Shell、文件、执行器能力纳入统一调用框架
- 用 SQLite、向量记忆与运行时配置覆盖形成持久层
- 用 FastAPI 提供稳定接口与 webhook 接入点
- 用 Reflex 提供 Web 控制台，而不是停留在 CLI 原型
- 用平台桥接适配 QQ、微信、飞书与 generic webhook
- 用治理与审计模块约束运行风险和关键操作轨迹

## 3. 核心能力概览

### 3.1 交互层

- CLI：适合开发者本地使用、排障、批处理与自动化脚本接入
- REST API：适合对外提供服务能力、供前端或第三方系统调用
- Reflex Web UI：适合配置、观察、治理与运行态操作
- 平台消息桥接：适合把 DeepCode 接入 QQ、微信、飞书等聊天平台

### 3.2 智能体层

- Orchestrator：负责任务拆解、阶段编排与状态推进
- Coder：负责代码生成与修改建议
- Reviewer：负责风险检查、质量审查与缺陷识别
- Tester：负责测试用例设计与验证反馈
- Reflection / Prompt Layers：负责提示词分层与推理上下文组织

### 3.3 工程运行层

- Tool Router：统一工具调度入口
- ShellTool：受控执行命令行能力
- FileManagerTool：文件读取、修改与管理
- Code Executor：执行代码、收集输出、处理超时和失败

### 3.4 平台与治理层

- Platform Bridge：统一平台消息标准化
- Platform Delivery：统一平台回发策略
- Policy Engine：策略校验与执行约束
- Approval Store / Audit：审批与审计记录
- Extension Registry：技能、Hook、MCP 注册与远程安装

## 5. 关键架构设计

### 5.1 为什么要做多智能体，而不是一个大 Prompt

因为软件工程任务天然具备阶段性和角色差异：

- 规划和执行的目标不同
- 编码和评审的判断标准不同
- 生成和验证的上下文结构不同

用一个大 Prompt 虽然实现快，但会带来三个问题：

- 上下文耦合严重，输出不稳定
- 难以观测每个阶段的中间状态
- 很难对不同阶段做独立优化和治理

DeepCode 选择多角色拆分，本质上是在用工程架构管理模型不确定性。

### 5.2 为什么要保留 CLI、API、Web UI 三种入口

因为三者分别解决不同场景：

- CLI 面向开发者效率和本地自动化
- API 面向系统集成和服务化
- Web UI 面向配置、可视化运维和非命令行用户

这不是重复建设，而是同一运行时能力在不同交互层的复用。

### 5.3 为什么平台桥接要做标准化层

QQ、微信、飞书的入站协议、签名方式、事件模型、回包格式差异很大。如果直接在业务逻辑里写平台分支，会迅速失控。

DeepCode 的做法是：

- 先把平台事件归一化成统一输入模型
- 再复用 chat runtime 和本地命令体系
- 最后按平台选择不同 delivery strategy 回发

这样带来的价值是：

- 业务逻辑只写一次
- 平台适配与核心能力解耦
- 后续新增平台成本更低

## 6. NapCat 集成设计亮点

### 6.3 工程价值

这部分能力说明项目已经从“本地 AI 编程助手”进化为“可接入外部业务渠道的 LLM Agent 服务”。这对于面试大模型应用开发岗位是一个明显加分点，因为它体现了：

- 协议适配能力
- 消息系统工程经验
- 异步回调链路设计能力
- 安全校验与幂等控制意识

面试可强调：

### 7.4 deepcode/governance

面试可强调：

- 大模型应用不能只看效果，还要考虑风险控制

## 8. 项目中的工程难点

### 8.1 大模型运行时与平台回调耦合问题

挑战：平台事件是异步入站，模型调用可能慢、失败、限流或被配置污染。

处理方式：

- 用运行时配置覆盖机制持久化关键模型参数
- 在平台异步回调链路中做空回复兜底
- 区分入站处理和出站投递，降低故障传播范围

### 8.2 Web UI 与后端运行时一致性

挑战：UI 中展示的模型配置与实际执行模型可能漂移。

处理方式：

- 配置保存后立即同步运行时覆盖
- 必要时清理缓存，确保请求命中最新配置

### 8.3 跨平台兼容

挑战：Windows、Conda、Reflex、Node/Bun、Shell 命令集兼容问题较多。

处理方式：

- 做 preflight / doctor 检查
- 保留 Python 模块入口兜底
- 对 Windows 命令行为、路径、timeout、环境变量做兼容修复

### 8.4 平台安全与幂等

挑战：Webhook 场景需要处理伪造请求、重复投递、签名与 token 校验。

处理方式：

- 引入签名校验与 token 校验机制
- 做 event_id 级别幂等处理
- 在日志与审计中保留关键轨迹

## 11. 面试官视角高频问题与参考答案

下面的问题按“项目理解 -> 架构设计 -> 工程实现 -> 风险治理 -> 平台接入 -> 优化演进”分层整理。

### 问题 1：你为什么要做这个项目？

参考答案：

我想做的不是一个聊天机器人 Demo，而是一个真正具备工程能力的大模型应用系统。软件工程场景天然需要任务拆解、工具调用、状态保存、执行验证和风险治理，所以我把项目设计成多智能体协作架构，并且同时提供 CLI、API、Web UI 和平台消息入口，验证它在不同交互场景下都能工作。

### 问题 2：这个项目和普通的 AI Chat 应用本质区别是什么？

参考答案：

普通 Chat 应用核心是“输入一段话，返回一段话”，而 DeepCode 的核心是“围绕任务闭环组织模型能力”。它有智能体编排、工具路由、持久化、平台桥接和治理能力，本质上更接近一个 AI runtime，而不是一个对话页面。

### 问题 3：为什么要采用多智能体而不是单 Agent？

参考答案：

因为复杂软件任务包含规划、实现、评审、测试多个阶段，不同阶段的目标函数不同。多智能体拆分后，系统的可观测性、可扩展性和可调优性更强，也更容易在某个阶段单独做 Prompt、工具权限和质量策略优化。

### 问题 4：多智能体会不会带来额外开销？

参考答案：

会，主要是 token 开销、上下文组织成本和系统复杂度上升。但换来的收益是结构化流程、阶段控制和质量稳定性。对于复杂工程任务，这种额外成本通常是值得的；对于简单问答，则可以走轻量模式。

### 问题 5：你怎么定义 Orchestrator 的职责边界？

参考答案：

Orchestrator 负责任务拆解、阶段调度和状态推进，不直接承载所有业务细节。具体执行由各子角色和工具完成。这样做的好处是把“流程控制”和“内容生成”解耦，避免一个组件既负责决策又负责执行，导致不可维护。

### 问题 6：工具调用为什么需要单独做 Router？

参考答案：

工具一旦变多，如果由 Agent 直接感知全部细节，耦合会非常高。Tool Router 的作用是统一参数校验、分发逻辑和失败兜底，这样新增工具时不需要改大量 Agent 主流程代码。

### 问题 7：ShellTool 为什么要做命令白名单？

参考答案：

因为大模型应用的一个核心风险就是越权执行。命令白名单是最基础的安全边界，至少能把工具调用限制在受控命令集合里，再结合 timeout、日志和审计，才能形成可接受的执行模型。

### 问题 8：为什么既有 CLI 又有 API 又有 Web UI？

参考答案：

这是典型的多入口复用同一套核心运行时能力。CLI 面向开发者效率，API 面向服务化集成，Web UI 面向可视化运维和配置。核心逻辑不应该和交互层绑死，否则项目很难成长为平台。

### 问题 9：为什么前端选择 Reflex？

参考答案：

这个项目主要能力在 Python 运行时，Reflex 可以让状态、事件和后端逻辑保持较强一致性，减少前后端重复建模的成本。对于需要快速构建运维型控制台的项目，这是一个很务实的选择。

### 问题 10：你为什么明确放弃 Streamlit 作为主路径？

参考答案：

因为项目已经从早期 Demo 进入到具备复杂状态和运行控制需求的阶段，Reflex 更适合承载可交互、状态驱动、结构化页面。Streamlit 更适合快速演示，但不适合作为长期演进的主运行时。

### 问题 11：平台桥接这一层为什么重要？

参考答案：

企业实际使用大模型时，很多交互不是发生在独立网页里，而是在 IM、工单、机器人或业务系统里。平台桥接层让核心 Agent 能力脱离单一交互界面，真正进入业务流程。

### 问题 12：QQ / NapCat 集成的工程难点是什么？

参考答案：

难点主要在协议差异、回调安全、消息回发格式和链路排障。NapCat 既有入站 webhook，又有 OneBot 风格的出站发送接口，且要考虑 token、消息段格式、媒体链接、监听端口和独立进程管理，所以不能只停留在“收消息后 print 一下”的程度。

### 问题 13：为什么要做独立 NapCat inbound listener，而不是都挂在主服务里？

参考答案：

独立 listener 的价值是部署灵活。某些场景下只需要平台接入能力，不需要把整套 API 服务暴露出来；另外独立进程也利于把平台入口和核心服务做进程级隔离，便于排障和部署编排。

### 问题 14：平台 webhook 怎么保证安全？

参考答案：

主要有三层：

- token 或签名校验，防止伪造请求
- 时间窗口控制，降低重放攻击风险
- event_id 幂等处理，避免重复投递导致多次执行

### 问题 15：为什么还要做审计和治理，不是把功能做出来就行吗？

参考答案：

大模型应用一旦具备工具执行能力，风险就已经进入工程层面了。治理和审计不是加分项，而是从“能跑”进入“可上线、可追溯、可控”的必要条件。

### 问题 16：你们的状态持久化为什么选择 SQLite？

参考答案：

因为当前阶段目标是本地优先、单机可运行、开发成本可控。SQLite 足够支撑会话、任务、配置和审计等结构化数据，而且易于发布和安装。后续如果面向多实例部署，再迁移到更重的数据库也比较顺。

### 问题 17：向量记忆在这个项目里起什么作用？

参考答案：

它主要解决长期语义上下文检索问题，不适合替代结构化状态。结构化状态交给数据库，语义记忆交给向量存储，两者职责不同。这样设计比“所有东西都丢给向量库”更合理。

### 问题 18：你如何处理 UI 配置和运行时配置不一致的问题？

参考答案：

关键思路是把模型配置持久化后立即同步到运行时覆盖层，并在必要时清理缓存，确保下一次调用命中最新配置。否则很容易出现“界面显示 A 模型，实际请求打到 B 模型”的漂移问题。

### 问题 19：为什么要保留 mock provider？

参考答案：

因为大模型项目如果所有验证都依赖真实 API，会导致测试成本高、排障慢、无法稳定复现。mock provider 可以让基础链路、状态机和 UI 流程在没有外部模型依赖时也能跑通。

### 问题 20：这个项目怎么做可测试性设计？

参考答案：

我尽量把平台输入标准化、把核心逻辑抽到可复用模块、把外部依赖通过 provider 或 store 抽象隔离出来，这样单元测试可以围绕路由、状态、平台适配、配置和存储分别验证，而不是只能做大而全的端到端测试。

### 问题 21：你做过哪些跨平台兼容处理？

参考答案：

重点处理了 Windows 下的命令执行、路径展开、Conda 环境差异、Reflex 依赖与 Bun/NPM 行为差异，以及 CLI 启动入口兜底。这些看似细碎，但决定了项目能不能真的被别人跑起来。

### 问题 22：如果让你继续迭代 0.3，你会优先做什么？

参考答案：

我会优先做三件事：

- 把任务执行链路进一步可观测化，增加更细粒度的阶段事件和 tracing
- 完善平台桥接的多租户与权限模型
- 强化插件生态，让技能、工具和策略可以更标准化地安装和治理

### 问题 23：如果面向生产，你觉得当前最大的风险是什么？

参考答案：

最大的风险不是模型能力本身，而是执行风险和配置复杂度。包括工具越权、平台回调误配置、运行时配置漂移、外部模型限流和长链路排障难度。这也是为什么我在 0.2 里重点补了治理、持久化和平台日志链路。

### 问题 24：这个项目最能体现你什么能力？

参考答案：

最能体现的是我把大模型能力工程化落地的能力，而不是只会调 API。包括架构拆分、协议适配、状态设计、治理思维、运行态排障和发布分发，这些组合在一起，才是一个完整的大模型应用开发能力。

### 问题 25：如果面试官质疑这是“功能堆砌”，你怎么回应？

参考答案：

我会从系统边界解释这些模块为什么存在。CLI、API、Web UI、平台桥接并不是互相独立的产品，而是围绕同一套 Agent runtime 的不同入口；治理、存储、扩展也不是装饰，而是让这个系统从 Demo 走向可运行平台的必要基础设施。

### 问题 26：为什么你认为这个项目适合写进“大模型应用开发”方向简历？

参考答案：

因为它覆盖了这个岗位最看重的几个维度：模型接入、Agent 架构、工具调用、服务化接口、平台接入、运行治理、部署发布和问题排查。相比单点 Demo，它更能证明候选人理解大模型应用的完整工程面。

### 问题 27：如何向没有做过 Agent 的面试官解释这个项目？

参考答案：

可以类比成“给大模型加了一个软件工程操作系统”。模型不是直接裸输出，而是被放进一个具备状态、工具、路由、策略和交互入口的运行时里。这样它才从一个聊天能力，变成一个可以完成复杂工程任务的系统。

### 问题 28：如何向很懂系统设计的面试官解释这个项目？

参考答案：

重点讲边界和抽象：Agent 编排层、工具调度层、平台标准化层、持久化层、治理层、交互层分别解决什么问题，为什么要解耦，以及这些层之间如何通过清晰的契约交互。对方通常更看重架构完整性和演进空间，而不是某个花哨功能。

## 12. 追问型深挖问题

下面这组问题更适合二面、交叉面或技术主管面。

### 深挖 1：如果并发平台事件很多，如何防止同一个会话被多个请求同时污染？

参考答案：

可以从三层处理：

- 存储层加会话级并发控制或版本控制
- 事件层以 message_id / event_id 做幂等和去重
- 运行时把会话写入和长耗时模型调用拆开，避免大锁包住整条链路

### 深挖 2：如果平台消息中混入恶意 prompt 或指令注入怎么办？

参考答案：

不能只依赖 Prompt 防御，应该同时做：

- 本地命令和工具参数白名单
- 角色边界和权限隔离
- 对外部输入与内部系统提示分层
- 对高风险动作走审批或策略校验

### 深挖 3：为什么审计日志不能只打在应用日志里？

参考答案：

因为应用日志偏排障，审计日志偏行为追溯和合规，两者目的不同。审计应该具备更稳定的结构、字段和留存策略，不能依赖临时日志输出。

### 深挖 4：如果要支持 SaaS 化，多租户怎么改？

参考答案：

至少要把以下边界租户化：

- 配置与模型凭据
- 会话、任务、产物和记忆
- 策略、审批和审计
- 平台 webhook 路由与签名密钥

同时需要在 API、Web UI、运行时覆盖和扩展安装中增加租户隔离。

### 深挖 5：如果要把 NapCat 扩展到更多消息平台，哪些抽象已经可以复用？

参考答案：

可复用的主要是：

- 平台入站标准化模型
- chat runtime 复用
- 本地命令体系
- delivery strategy 抽象
- 安全校验与幂等处理框架

这说明当前设计不是只为 NapCat 写死的，而是先抽象出模式，再落地一个具体平台。

## 13. 面试表达建议

### 13.1 讲项目时的推荐顺序

建议按这个顺序讲：

1. 先讲项目目标和场景
2. 再讲整体架构和多入口设计
3. 再讲一个你最有代表性的技术难点，例如 NapCat 接入或运行时一致性
4. 最后讲工程化补齐项，例如治理、测试、发布、Docker

### 13.2 不要只讲“用了什么技术”

更重要的是讲：

- 为什么这么设计
- 解决了什么问题
- 做过哪些权衡
- 遇到什么坑、怎么修
- 如果继续做，下一步会怎么演进

### 13.3 如果时间很短

可以压缩成一段 60 秒表达：

我做了一个叫 DeepCode 的大模型应用项目，它不是普通聊天机器人，而是一个支持多智能体协作、工具调用、平台桥接和治理审计的工程智能体系统。后端基于 FastAPI 和 Python，前端用 Reflex，支持 CLI、REST API、Web UI 和 QQ/NapCat 接入。我重点解决了平台 webhook 标准化、NapCat 入站与回发、运行时配置一致性、工具安全边界和发布分发这些问题，所以这个项目比较完整地体现了大模型应用工程化落地能力。

## 14. 结论

DeepCode 0.2 最大的价值，不在于“接了多少模型”，而在于它已经具备一个大模型应用平台应有的骨架：

- 有任务编排
- 有工具体系
- 有状态和记忆
- 有服务接口
- 有平台接入
- 有治理审计
- 有 UI 运维
- 有发布分发

对于大模型应用开发岗位，这样的项目更容易证明候选人真正理解的是“如何把模型能力做成系统”，而不是只会调用一次 API。
